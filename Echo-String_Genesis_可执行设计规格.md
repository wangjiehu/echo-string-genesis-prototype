# 《语弦生态：创世纪》可执行设计规格

版本：v1.0
定位：从高概念方案收束为可原型、可测试、可迭代的游戏设计规格

当前状态：已建立MVP数据表、灰盒关卡文档、任务与测试文档，并完成一个可浏览器运行的静态灰盒交互原型。1.0.0原型已从Z1跑通到Z7，用于验证扫描、剥离、注入、组合反应、力场谜题和Boss护文闭环；同时保留网页直玩、引导/低引导/调试三模式、状态仪表、对象生命与护文反馈、词库写入路由、事件日志、通关路径判定、质量评分、路线断言、软锁审计、区域检查点恢复、玩家可见路径审计、真人试玩反馈事件、真实DOM点击回归和桌面/手机视觉Smoke。

## 0. 本轮审查结论

现有方案的核心创意成立，但如果按“完整3A开放世界 + 自由语义改写 + 大规模文明语系切换”同时推进，生产风险会过高。真正需要先证明的不是世界能有多大，而是玩家在十几分钟内能否理解并享受“扫描、取词、改写、反应”的基本乐趣。

本轮直接做以下优化：

1. **取消首版自由文本输入**：玩家不能任意打字造词，首版只使用设计好的词缀库。
2. **引入槽位系统**：每个对象只开放少量可改写槽位，避免所有词缀对所有物体生效。
3. **推迟因果闭环**：复杂的主谓宾关系链不进入MVP，先用单体词缀改写和少量组合反应证明核心。
4. **推迟文明语系切换**：文明语系保留为中后期系统，MVP不承载它。
5. **重写MVP范围**：明确第一个可玩切片的场景、词缀、敌人、谜题、Boss和验收标准。
6. **强化失败反馈**：玩家改写失败时必须知道原因，而不是猜系统边界。
7. **技术实现降风险**：先做数据驱动的局部对象系统，不做全局实时开放世界语义模拟。

## 1. 核心定位

### 1.1 一句话概念

玩家扮演能够读取并改写物体“语义槽位”的异墨者，在崩坏世界中通过剥离、注入和组合词缀改变物理、敌人和关卡条件，修复失控的叙事生态。

### 1.2 类型

- 第三人称动作冒险
- 生态解谜
- 沉浸式模拟倾向
- 单人剧情驱动

首个可玩版本不做开放世界，而做一个密度高、规则清楚、可反复测试的垂直切片。

### 1.3 玩家幻想

玩家不是无所不能的作者，而是带着有限工具进入故障文本的“现场修正文员”。这比“神一样随便改世界”更适合游戏，因为限制会制造判断、代价和解法。

玩家应感受到：

- 我能看见别人看不见的底层规则。
- 我能把环境、敌人和机关改成自己的工具。
- 我造成的变化有清晰逻辑，不是随机魔法。

## 2. 设计支柱

### 2.1 可读性高于自由度

玩家必须先读懂规则，才会享受改写。任何词缀效果都要能通过UI、特效、声音和物理变化被快速理解。

### 2.2 改写必须有代价

每次改写消耗“原墨”，且词缀有持续时间、作用范围和槽位限制。没有代价的改写会迅速变成万能钥匙。

### 2.3 少量词缀，多种用途

首版不追求词缀数量，而追求同一词缀在战斗、解谜和探索中有不同价值。例如“轻盈”既能移动巨石，也能让敌人被击飞更远。

### 2.4 惊喜必须可复盘

组合反应可以惊艳，但不能像黑箱。玩家事后应能说清楚：“因为我把过热和潮湿放到一起，所以产生蒸汽爆发。”

### 2.5 关卡负责教学

不要靠长说明教玩家。每个区域用安全对象、低压敌人和明确障碍逐步教会一个规则。

## 3. 世界观收束

### 3.1 基础设定

世界由“原初作者”写成。作者停笔后，世界失去维护，文本开始自我增殖、互相矛盾和局部缺页。玩家角色“异墨者 · 楷”体内流淌空白原墨，可以临时修正局部语义。

### 3.2 三类灾害

| 灾害 | 本质 | 游戏表现 |
| --- | --- | --- |
| 重词污染 | 某个词缀过量复制 | 火海、永冻区、增殖森林、重力塌陷 |
| 语义冲突 | 两个互斥词缀同时存在 | 燃烧又冻结、死亡又行动、坚固又崩裂 |
| 文本缺页 | 对象缺少必要描述 | 无碰撞地面、断裂路径、乱码空间 |

### 3.3 主线方向

主角的目标不是恢复旧秩序这么简单，而是逐步发现：世界并不一定需要原初作者继续支配。后期主题可以转向“修复旧文本”与“允许新文明自写规则”的冲突。

MVP只呈现开端：主角在“断句庭院”醒来，学习最基础的读取和改写能力，进入修正文库。

## 4. 语义系统规格

### 4.1 对象结构

世界中只有“语义对象”可以被改写。每个语义对象由以下数据组成：

| 字段 | 说明 |
| --- | --- |
| 对象名词 | 门、石像、敌人、巨石、水池、火盆 |
| 基础状态 | 对象默认属性，不一定可剥离 |
| 开放槽位 | 玩家可以改写的词缀槽 |
| 锁定槽位 | 剧情或Boss机制保护的槽 |
| 当前词缀 | 槽位上正在生效的词 |
| 抗性 | 对某类词缀的接受或拒绝 |
| 反应表 | 与其他词缀共存时触发的组合反应 |

### 4.2 词缀槽位

首版使用固定槽位，而不是让任意词缀随便叠加。

| 槽位 | 可承载词缀 | 影响范围 |
| --- | --- | --- |
| 结构槽 | 坚固、脆弱、封闭 | 血量、破坏、通行状态 |
| 质量槽 | 沉重、轻盈 | 质量、击退、移动、平台用途 |
| 温度槽 | 过热、冻结 | 伤害、控制、环境状态 |
| 介质槽 | 潮湿 | 元素反应、传导、蒸汽 |
| 力场槽 | 吸引、排斥 | 拉扯、冲击、聚怪 |
| 行为槽 | 守卫、追猎、逃离 | 简单AI倾向，MVP只读不可改写 |

### 4.3 词缀规则

1. 同一槽位同一时间只允许一个主词缀。
2. 新词缀写入同槽位时，会替换旧词缀，而不是无限叠加。
3. 某些对象只开放部分槽位。例如门可以有结构槽，但没有行为槽。
4. Boss和关键机关可以有锁定槽位，但必须显示锁定原因。
5. 区域规则优先级高于单体改写。例如“禁热区”会压制过热效果。
6. MVP开放给玩家改写的槽位为结构、质量、温度、介质和力场；行为槽只用于敌人状态展示与AI调参。

### 4.4 解析优先级

语义解析按固定顺序执行：

1. 区域规则
2. 对象锁定槽位
3. 玩家临时改写
4. 对象基础状态
5. 互斥清理
6. 组合反应
7. 物理、材质、AI、特效应用

这套顺序能减少“为什么我明明改了却没生效”的困惑。

### 4.5 操作方式

| 操作 | 作用 | MVP状态 |
| --- | --- | --- |
| 扫描 | 显示目标语义槽位和可改写状态 | 必做 |
| 剥离 | 从目标开放槽位取出词缀，存入临时词库 | 必做 |
| 注入 | 将词库中的词缀写入目标开放槽位 | 必做 |
| 覆写 | 用新词替换同槽旧词 | 必做 |
| 因果闭环 | 建立对象之间的主谓宾关系 | 后期 |
| 文明语系切换 | 改变语义解释规则 | 后期 |

### 4.6 资源限制

玩家使用“原墨”进行改写。

- 扫描不消耗资源。
- 剥离少量消耗原墨。
- 注入中等消耗原墨。
- 组合反应额外消耗原墨或稳定度。
- 原墨可通过战斗、环境节点或存档点恢复。

同时加入“临时词库容量”：MVP中玩家最多携带3个临时词缀。这样能迫使玩家选择，而不是囤积所有解法。

## 5. 核心玩法循环

### 5.1 标准循环

1. 观察障碍或敌人。
2. 扫描，读取语义槽位。
3. 判断缺少哪个词缀或需要移除哪个词缀。
4. 从环境或敌人身上剥离目标词缀。
5. 注入到关键对象。
6. 触发物理变化、通路变化或战斗优势。
7. 进入下一组更复杂的组合问题。

### 5.2 战斗循环

1. 普通攻击、闪避和弹反制造破绽。
2. 敌人破绽时显示核心槽位。
3. 玩家剥离防御、质量或温度词缀。
4. 玩家注入相克词缀或利用环境反应。
5. 敌人进入新状态，招式或抗性改变。

### 5.3 探索循环

探索重点不是找钥匙，而是找“可用词”。玩家看到障碍后，会回忆附近哪里存在对应词缀。

例子：

- 门是封闭且坚固的，附近石像有脆弱。
- 巨石太沉，附近浮碑有轻盈。
- 敌人有过热，水池有潮湿，两者结合可制造蒸汽。

## 6. MVP垂直切片

### 6.1 场景名称

**断句庭院**

定位：15到20分钟的可玩切片，用一个小区域完整验证语义改写是否成立。

### 6.2 场景目标

玩家从庭院入口醒来，穿过被封闭、冻结和重力异常影响的区域，击败“残章守卫”，进入原初修正文库。

### 6.3 MVP不做内容

- 不做开放世界。
- 不做文明语系切换。
- 不做自由输入文字。
- 不做复杂因果闭环。
- 不做真实流体模拟。
- 不做大量装备和等级成长。

### 6.4 MVP词缀表

| 词缀 | 槽位 | 主要效果 | 解谜用途 | 战斗用途 |
| --- | --- | --- | --- | --- |
| 封闭 | 结构槽 | 阻止通行或开启 | 构成门和机关障碍 | 可作为敌人护盾状态 |
| 坚固 | 结构槽 | 提高耐久和抗击退 | 保护机关或障碍 | 敌人减伤 |
| 脆弱 | 结构槽 | 降低耐久，易破坏 | 打碎门、墙、石像 | 打出高伤窗口 |
| 沉重 | 质量槽 | 增加质量和压制力 | 压住机关、阻挡移动 | 敌人冲撞更强但更慢 |
| 轻盈 | 质量槽 | 降低质量和重力影响 | 移动巨石、制造平台 | 敌人更易被击飞 |
| 过热 | 温度槽 | 持续热伤和点燃倾向 | 融化冰封物 | 增加伤害但可被反应 |
| 冻结 | 温度槽 | 锁定移动或降低速度 | 固定平台、停住机关 | 控制敌人 |
| 潮湿 | 介质槽 | 提高导热和蒸汽反应 | 灭火、导热、触发蒸汽 | 削弱过热敌人 |
| 吸引 | 力场槽 | 将附近对象拉向中心 | 聚拢碎片或平台 | 聚怪 |
| 排斥 | 力场槽 | 向外推开对象 | 推动机关、打开间隙 | 击退群体 |

### 6.5 MVP组合反应

| 组合 | 反应 | 效果 |
| --- | --- | --- |
| 过热 + 潮湿 | 蒸汽爆发 | 产生遮蔽、击退和短暂致盲 |
| 冻结 + 脆弱 | 破裂窗口 | 目标受到下一次重击时大幅破坏 |
| 吸引 + 排斥 | 力场震荡 | 先聚拢再爆开，适合群体控场 |

组合反应必须少而清楚。宁可三个反应做扎实，也不要十几个反应都像特效彩蛋。

### 6.6 MVP对象

| 对象 | 默认词缀 | 可交互点 |
| --- | --- | --- |
| 封文大门 | 封闭、坚固 | 注入脆弱后可破坏 |
| 裂纹石像 | 脆弱 | 提供玩家第一个可剥离词 |
| 压机关石 | 沉重 | 可压住地面机关 |
| 浮字碑 | 轻盈 | 提供移动巨石的解法 |
| 冻结水池 | 冻结、潮湿 | 可制造平台或蒸汽反应 |
| 火盆 | 过热 | 提供融冰和战斗反应词 |
| 力场柱 | 吸引或排斥 | 教学推拉和聚拢 |

### 6.7 MVP敌人

| 敌人 | 行为 | 语义特点 | 设计目的 |
| --- | --- | --- | --- |
| 字壳游兵 | 近战巡逻 | 可剥离坚固或脆弱 | 教会战斗扫描 |
| 缀刺兽 | 冲撞和扑击 | 沉重使冲撞强但转身慢 | 教会质量改写 |
| 残章守卫 | 小Boss | 阶段性锁定结构槽 | 验证Boss语义机制 |

### 6.8 MVP流程

1. **入口教学**：玩家扫描封文大门，看到“封闭”和“坚固”。
2. **第一次剥离**：玩家从裂纹石像剥离“脆弱”。
3. **第一次注入**：玩家给大门注入“脆弱”，破坏大门。
4. **质量谜题**：玩家从浮字碑获取“轻盈”，让压机关石可移动。
5. **第一次战斗**：字壳游兵出现，玩家通过攻击制造破绽并剥离“坚固”。
6. **元素反应教学**：玩家把“潮湿”和“过热”组合，触发蒸汽爆发。
7. **力场谜题**：玩家使用吸引和排斥移动碎片，形成通路。
8. **小Boss战**：残章守卫三层护文分别响应蒸汽爆发、破裂窗口和力场震荡，玩家必须轮换环境词缀反制。
9. **出口奖励**：玩家进入修正文库，看到更大的世界规则伏笔。

## 7. 战斗系统

### 7.1 基础动作

MVP只需要轻攻击、重攻击、闪避、格挡、完美弹反和织叶仪操作。动作系统要服务语义窗口，不追求复杂连招树。

### 7.2 语义破绽

敌人不会随时允许玩家改写核心槽位。玩家必须通过以下方式制造“语义破绽”：

- 完美弹反
- 重攻击破防
- 环境反应命中
- 敌人招式后摇

破绽期间，敌人的一个到两个核心槽位会高亮，玩家可以快速剥离或注入。这样能保持动作节奏，不让玩家每秒都停下来读界面。

### 7.3 弹反定位

原设想中“弹反后改写攻击句法宾语”很酷，但MVP中实现成本和可读性风险较高。首版弹反只做两件事：

1. 打断攻击。
2. 打开短暂语义破绽窗口。

“改写攻击对象”保留为后期高级技能。

### 7.4 敌人状态设计

敌人每次只展示最关键的两个语义状态。不要把完整数据库都显示给玩家。

例子：

- 字壳游兵：坚固、追猎
- 缀刺兽：沉重、过热
- 残章守卫：坚固锁定、温度可改写

## 8. 关卡设计规范

### 8.1 主解法与旁路

每个谜题必须有一个清晰主解法，允许一到两个旁路解法。旁路解法只能来自玩家已学过的词缀，不能靠猜隐藏规则。

### 8.2 防卡死原则

关键词缀来源必须可恢复：

- 石像被打碎后，脆弱词缀仍可从碎片中读取。
- 火盆熄灭后，附近有第二个过热点。
- 玩家丢失关键词缀后，可在区域节点重新提取。

### 8.3 锁定规则

若对象不能被改写，UI必须解释原因：

- 槽位不存在
- 槽位已锁定
- 词性不匹配
- 区域规则压制
- 原墨不足

### 8.4 区域规则

区域规则是关卡设计的“天花板”。它定义玩家在这个区域里可以玩什么，不能玩什么。

MVP的区域规则：

```text
[断句庭院] -> [允许] -> [结构、质量、温度、介质、力场改写]
[断句庭院] -> [禁止] -> [因果闭环、文明语系切换]
```

## 9. UI与交互

### 9.1 扫描界面

扫描时只显示三类信息：

1. 对象名称
2. 当前关键词缀
3. 可操作提示

示例：

```text
封文大门
结构槽：封闭 / 坚固
可注入：脆弱
```

不要把完整语义链铺满屏幕。视觉上应像战术读数，而不是文档弹窗。

### 9.2 词库界面

玩家最多携带3个临时词缀。词库以轮盘或横向短栏展示，每个词缀有颜色、图标和槽位标识。

### 9.3 预览反馈

注入前应显示短预览：

- 可生效：目标槽位亮起，显示预计变化。
- 会替换：旧词缀淡出，新词缀高亮。
- 不可生效：显示明确原因。

### 9.4 操作建议

键鼠：

- 右键长按：扫描
- E：剥离
- Q：打开词库
- 左键：注入选中词
- 空格：闪避
- 鼠标中键或F：格挡/弹反

手柄：

- LT：扫描
- X：剥离
- LB：词库轮盘
- RT：注入
- B：闪避
- Y：格挡/弹反

## 10. 美术与音频方向

### 10.1 视觉原则

文字不能像贴图字幕一样浮在表面，而要像物体内部的结构显影。平时世界保持实体感，扫描和改写时才让语义层浮现。

### 10.2 词缀视觉字典

| 类别 | 视觉方向 | 声音方向 |
| --- | --- | --- |
| 结构 | 裂纹、硬边、金属摩擦、字形断裂 | 石裂、金属压缩 |
| 质量 | 下坠尘环、漂浮墨粒、重低频 | 低频压迫或轻薄上扬 |
| 温度 | 热浪、冰晶、蒸汽、局部折射 | 灼烧声、冻结脆响 |
| 介质 | 水膜、湿痕、火星、扩散涟漪 | 潮湿滴落、燃烧噼啪 |
| 力场 | 环形波纹、拉伸字线、空间扭曲 | 吸入声、冲击波 |

### 10.3 特效预算

MVP所有高强度特效必须短促、可读、能暴露敌人轮廓。不要让文字粒子遮挡攻击前摇。

## 11. 技术落地

### 11.1 引擎选择

长期可以使用Unreal Engine 5，但MVP不依赖Nanite、Mass Entity或复杂流体模拟。核心风险在玩法系统，不在画面堆料。

### 11.2 推荐模块

| 模块 | 职责 |
| --- | --- |
| `SemanticEntityComponent` | 保存对象槽位、词缀、锁定状态和抗性 |
| `SemanticInventoryComponent` | 管理玩家临时词库和原墨 |
| `SemanticResolverSubsystem` | 统一处理替换、互斥、优先级和失败原因 |
| `SemanticEffectApplier` | 将解析结果应用到物理、材质、AI和特效 |
| `SemanticReactionSubsystem` | 处理少量策划定义的组合反应 |
| `SemanticScanWidget` | 显示扫描、预览和错误反馈 |
| `SemanticDebugOverlay` | 开发期显示槽位、优先级、反应日志 |

### 11.3 数据结构

词缀使用数据资产或表驱动：

| 字段 | 说明 |
| --- | --- |
| TagId | 唯一标识，例如 `Semantic.Structure.Fragile` |
| DisplayName | 本地化名称 |
| SlotType | 结构、质量、温度、介质、力场、行为 |
| Priority | 解析优先级 |
| Duration | 持续时间，永久或临时 |
| InkCost | 注入消耗 |
| ExtractCost | 剥离消耗 |
| CompatibleObjectTypes | 可作用对象类型 |
| PhysicsModifier | 质量、重力、速度、碰撞等 |
| MaterialModifier | 材质参数 |
| AiModifier | AI行为参数 |
| VfxId | 特效 |
| FailureText | 失败提示文本 |

### 11.4 反应数据

组合反应必须由策划显式配置，不做任意笛卡尔积推导。

| 字段 | 说明 |
| --- | --- |
| RequiredTags | 触发所需词缀 |
| RequiredSlotState | 目标或区域状态 |
| TriggerEvent | 注入、受击、计时、进入范围 |
| Effect | 伤害、击退、控制、开路 |
| Cooldown | 防止连锁爆炸 |
| VfxId | 反应特效 |

### 11.5 技术取舍

| 原设想 | MVP处理 |
| --- | --- |
| 真实流体模拟 | 用材质、粒子和碰撞盒假模拟 |
| 全世界实时语义重写 | 只对标记对象生效 |
| 大规模ECS生态 | 普通组件 + 子系统即可 |
| 任意词语组合 | 策划定义组合表 |
| 复杂句法链 | 槽位词缀系统 |

### 11.6 必备调试工具

没有调试工具，这个项目会很快失控。MVP必须带：

- 对象语义槽位可视化
- 当前解析优先级显示
- 词缀注入失败原因日志
- 组合反应触发日志
- 原墨消耗统计
- 每个谜题的完成路径记录

## 12. 可测试验收标准

MVP完成不以“内容做完”为标准，而以玩家测试结果为标准。

### 12.1 可理解性

- 新玩家在3分钟内能理解扫描界面。
- 新玩家在5分钟内能完成第一次剥离和注入。
- 玩家能说出至少一个组合反应的因果逻辑。

### 12.2 可玩性

- 断句庭院完整流程控制在15到20分钟。
- 至少有两个谜题存在非唯一解法。
- 小Boss至少需要玩家使用一次语义改写，而不是纯砍过去。

### 12.3 可读性

- 敌人攻击前摇不被UI和特效遮挡。
- 词缀替换结果在1秒内能被玩家看懂。
- 操作失败时，玩家能看到明确原因。

### 12.4 技术稳定性

- 关键谜题不能因词缀丢失卡死。
- 组合反应不能无限连锁。
- 场景内同屏10个语义对象时保持稳定帧率。

## 13. 后期系统保留

以下系统有价值，但不进入MVP。

### 13.1 因果闭环

后期可升级为“句法绑定”：

```text
[大门] -> [消耗] -> [火盆热量]
[敌人] -> [追逐] -> [噪声源]
```

它适合中后期谜题，但必须建立在玩家已经熟悉单体词缀改写之后。

### 13.2 文明语系切换

文明语系可以作为第二阶段核心扩展：

| 语系 | 作用 |
| --- | --- |
| 现代语义 | 精确、一对一、适合Boss和机关 |
| 古典文言 | 大范围、意象化、高风险 |
| 原始图腾 | 文字实体化、空间重构 |

首版只做现代语义，保证规则清楚。

### 13.3 开放区域

开放区域应在垂直切片验证通过后再设计。届时每个区域围绕一个核心灾害展开，而不是直接堆大地图。

## 14. 开发路线

### 14.1 纸面验证

- 完成10个MVP词缀的数据表。
- 完成3个组合反应规则。
- 画出断句庭院灰盒流程图。
- 写清每个谜题的主解法和旁路解法。

### 14.2 灰盒原型

- 实现扫描、剥离、注入。
- 实现词库容量和原墨消耗。
- 实现结构、质量、温度、介质、力场五类效果。
- 实现三个组合反应。
- 完成无美术灰盒流程。

### 14.3 战斗切片

- 实现字壳游兵和缀刺兽。
- 实现破绽窗口。
- 实现小Boss残章守卫。
- 调整战斗中UI出现频率。

### 14.4 视觉切片

- 完成语义显影材质。
- 完成词缀视觉字典。
- 完成三个组合反应特效。
- 完成断句庭院最终美术方向样张。

### 14.5 玩家测试

- 观察玩家是否会扫描。
- 观察玩家是否知道从哪里取词。
- 观察玩家是否滥用单一解法。
- 记录卡死点和误解点。
- 根据测试结果删减或重排规则。

## 15. 当前结论

《语弦生态：创世纪》最值得保留的不是“3A宏大叙事”，而是“语义槽位改写”这套可操作玩法。它能把战斗、解谜、探索和世界观统一起来，但前提是规则足够克制。

下一步不该继续扩世界，而该把当前静态灰盒原型继续压实：真实浏览器DOM点击脚本和桌面/手机视觉Smoke已经接入验证链路，后续优先做低引导真人试玩，基于每区耗时、误操作报告和玩家复述调整教学节奏。只要这个切片能让玩家自然理解取词、改写和组合反应，项目就有继续扩展的价值；如果这个切片不好玩，再大的世界观也救不了它。
